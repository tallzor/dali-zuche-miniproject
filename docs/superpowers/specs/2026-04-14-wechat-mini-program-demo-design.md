# 微信小程序甲方展示 Demo 设计说明

## 摘要

本设计说明定义一个用于甲方展示的 `微信小程序风格移动端 Demo`。  
目标不是交付正式生产版本，而是做出一个“看起来接近可落地 MVP”的业务样机，用于展示本地深度自驾租车产品的核心产品结构、关键业务流程和后续可运营性。

本次 Demo 只做前台展示载体，不做真实后台系统，但会在小程序内提供一个 `隐藏运营演示入口`，用于展示商家录车、车辆资料上传、GPS 绑定和车辆上架流程。

## 目标与成功标准

### 目标

- 做出一个可本地运行的移动端 Web Demo
- 视觉和交互风格接近微信小程序
- 满足甲方现场演示和方案讲解
- 同时覆盖游客端租车主流程和商家录车上架流程

### 成功标准

- 甲方能从首页顺畅演示到租车申请、风控、支付成功、待交付
- 演示中可以切换关键异常状态，不只展示成功路径
- 可以进入隐藏运营入口，看到门店、车型模板、具体车辆、GPS 绑定和上架结果
- Demo 整体观感像真实产品，而不是纯视觉原型
- 所有流程用本地 mock 数据驱动，不依赖后端和联网能力

## 范围定义

### 本次要做

- 游客端页面和业务主流程
- 隐藏运营演示入口
- 本地 mock 数据和状态切换
- 小程序风格移动端视觉系统
- 关键异常页面

### 本次不做

- 真实微信小程序工程
- 真实后台系统
- 真实支付、风控、上传、GPS 接口
- 真实用户认证
- 多商家入驻能力
- 异地环线取还网络
- 个性化路线规划系统

## 产品结构

Demo 包含两条主线：

### 一、游客端主线

- 首页
- 车型列表
- 车型详情
- 租车申请
- 实名认证与驾驶证上传
- 风控结果
- 支付成功与待交付
- 订单详情
- 路线卡展示页
- 异常状态页

### 二、隐藏运营演示主线

- 运营入口页
- 门店管理页
- 车型模板页
- 录入具体车辆页
- 车辆资料上传结果页
- GPS 绑定结果页
- 车辆上架状态页
- 车辆列表页

## 视觉方向

本次 Demo 采用 `标准可信型` 视觉方向。

### 视觉关键词

- 标准
- 透明
- 可信赖
- 接近成品
- 业务感强

### 视觉原则

- 首页先讲业务承诺，再讲内容和路线
- 用深海军蓝、留白、浅灰蓝做主色系统
- 用安全绿表示通过状态，用提醒橙表示异常或说明状态
- 全局统一使用卡片化信息结构、清晰表单和明确状态标签
- 避免旅游海报风和过强营销感

## 页面设计

## 1. 首页

### 目的

在 3 秒内传达产品定位和核心价值。

### 必须展示

- 产品名称与口号
- 四个核心承诺：价格透明、标准交付、风控前置、本地路线服务
- 主按钮：立即选车
- 次级入口：查看路线、查看订单
- 异常状态切换入口
- 隐藏运营入口提示位，但不抢主视觉

## 2. 车型列表页

### 目的

展示“按场景选车”而不是“车海展示”。

### 必须展示

- 三类车型：
  - 情侣轻松出游
  - 家庭舒适出行
  - 远线稳妥出行
- 每个车型卡片展示：
  - 日租价格
  - 适合人群
  - 车辆亮点
  - 可租状态标签

## 3. 车型详情页

### 目的

增强信任感和下单意愿。

### 必须展示

- 车型主图
- 核心亮点
- 适合路线
- 费用包含和不包含
- 可租日期区间
- 进入申请页按钮

## 4. 租车申请页

### 目的

展示标准化申请流程和核心订单字段。

### 必须展示

- 用车日期
- 取车门店
- 出行人数
- 联系方式
- 增值项选择
- 提交申请按钮

## 5. 实名认证与驾驶证上传页

### 目的

把风控前置逻辑展示清楚。

### 必须展示

- 身份信息输入区
- 驾驶证上传区
- 提交审核按钮
- 风控说明提示

## 6. 风控结果页

### 必须展示两种状态

- 风控通过：进入支付
- 风控拒绝：给出友好提示和客服入口

### 设计要求

- 通过态强调业务闭环进入下一步
- 拒绝态强调规则清晰和可联系人工协助

## 7. 支付成功与待交付页

### 目的

展示支付完成后的订单承接和待交付状态。

### 必须展示

- 订单号
- 车型信息
- 取车时间
- 门店信息
- 当前状态：已支付待交付
- 客服即将联系提示
- 路线卡预览入口

## 8. 订单详情页

### 目的

展示订单后续承接能力和规则透明性。

### 必须展示

- 订单状态时间线
- 费用明细
- 退款规则入口
- 客服入口

## 9. 路线卡展示页

### 目的

把“本地深度服务”具体化。

### 必须展示

- 静态路线图
- 推荐停靠点
- 适合人群
- 路况提示
- 停车提示
- 车内附纸质路线卡说明

## 10. 异常状态页

### 本次只做轻量异常

- 风控拒绝
- 暂无可约
- 退款规则说明

### 原则

- 页面要能从主流程快速切换进入
- 不做复杂分支逻辑，只做稳定演示态

## 11. 运营入口页

### 目的

在不做正式后台的前提下，让甲方看到商家侧的管理思路。

### 必须展示

- 门店
- 车型模板
- 具体车辆
- GPS
- 上架状态

## 12. 门店管理页

### 必须展示

- 门店列表
- 每个门店的可交付车辆数
- 联系人
- 营业时间

## 13. 车型模板页

### 必须展示

- 模板名称
- 适合人群
- 默认日租价
- 已绑定车辆数量

## 14. 录入具体车辆页

### 必须展示

- 所属门店
- 所属车型模板
- 车牌号
- VIN
- 当前里程
- 保险到期
- 当前车况标签

## 15. 车辆资料上传结果页

### 必须展示

- 行驶证已上传
- 保险信息已上传
- 车辆实拍已上传
- 划痕记录已上传
- 保养摘要已录入

## 16. GPS 绑定结果页

### 必须展示

- 设备编号
- 当前在线状态
- 最近上报时间
- 已绑定车辆信息

## 17. 车辆上架状态页

### 必须展示

- 状态流转：
  - 草稿
  - 待检查
  - 已上架可租
- 一条明确说明：该车辆已可分配给订单

## 18. 车辆列表页

### 必须展示

- 车辆名称
- 所属门店
- 当前状态
- GPS 状态
- 是否可租

## 数据策略

所有数据使用本地 mock。

### 固定 mock 范围

- 2 个门店
- 3 个车型分类
- 3 台具体车辆
- 2 条路线卡
- 2 套订单结果
- 2 套风控结果

### 原则

- 不做动态生成
- 不做随机数据
- 所有演示内容稳定可复现

## 交互策略

### 必须支持

- 页面跳转
- 表单填写体验
- 状态切换
- 异常页切换
- 隐藏运营入口进入和返回

### 不做

- 真支付
- 真上传
- 真风控接口
- 真定位地图
- 真消息通知

## 技术方案

采用 `React + Vite` 做移动端 Web Demo。

### 原因

- 适合从 0 快速搭建多页面业务样机
- 容易控制移动端小程序风格界面
- 方便使用本地 mock 数据和路由状态
- 后续若决定继续推进，可再评估迁移到真实小程序工程

### 技术边界

- 单页面应用
- 多页面路由
- 轻量 CSS 样式
- 不引入重型后台 UI 框架

## 演示节奏

甲方演示建议按固定顺序走：

1. 首页讲业务定位
2. 车型列表讲按场景选车
3. 申请页讲标准下单
4. 实名与驾驶证页讲风控前置
5. 风控通过与拒绝讲业务可控
6. 支付成功和待交付页讲交易闭环
7. 路线卡页讲本地服务差异化
8. 隐藏运营入口讲录车、GPS 和上架
9. 最后切一个异常页，证明系统考虑了真实业务场景

## 验收标准

- 首页能清晰表达“标准可信型”价值主张
- 主流程可以从首页演示到支付成功
- 风控通过和拒绝都能稳定切换展示
- 暂无可约和退款说明都能单独展示
- 隐藏运营入口可进入并完成录车上架演示
- 整体视觉统一，像同一个产品，而不是页面拼贴
- 不依赖后端即可本地稳定运行

## 默认假设

- 本次仅做甲方展示 Demo，不做正式生产交付
- 本次仅做 Web 技术实现的小程序风格演示版
- 本次不做正式后台，而是通过隐藏运营入口演示商家录车与车辆上架逻辑
- 本次以主成功流程为主，同时补充少量关键异常态
- 录车和上架流程以演示结果页为主，不做复杂校验和真实数据持久化
